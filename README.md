# Personal AI Ops Platform

Self-hosted foundation for an always-on AI brain that lives on your VPS, talks to you through Telegram, and coordinates named agents across your office PC, home PC, and cloud workers.

## Clear architecture

### 1. Ollama layer

- The [`ollama`](docker-compose.yml) service runs the official Ollama container.
- Ollama is installed inside Docker, not directly on your VPS host OS.
- Model files are stored in the persistent Docker volume `ollama_data`.
- On first startup, [`docker/ollama/entrypoint.sh`](docker/ollama/entrypoint.sh) pulls every model listed in `OLLAMA_MODELS`.
- `OLLAMA_MODEL` is the default active model at first boot.

### 2. Runtime model selection

- The active chat/report model is stored in SQLite, so it survives container restarts.
- You can switch the active model later from:
  - Telegram with `/usemodel`
  - Your app through `POST /api/v1/models/select`
- Available model information is exposed through `GET /api/v1/models`.
- If you switch to an Ollama model that is not installed yet, the app can pull it automatically.

### 3. Telegram layer

- The Telegram bot runs as the [`bot`](docker-compose.yml) service.
- It uses long polling, so you do not need to set up a webhook or reverse proxy just for Telegram.
- It accepts:
  - normal chat messages
  - task commands
  - model-management commands

### 4. Reporting layer

- Daily reports are generated by Celery Beat.
- The report always includes structured task and agent counts.
- The job also tries to generate a short AI summary using the currently active model.
- If the LLM fails, the report still exists using the structured fallback text.

### 5. Device control layer

- A browser control panel is available at `/control`
- Installed agents can receive direct control commands in addition to background tasks
- Supported control capabilities now include:
  - screenshot capture for live-view polling
  - mouse move, click, and drag
  - keyboard typing, key press, and hotkeys
  - file browsing and file reading
  - process listing and process termination
  - window listing and focus
  - app launch plus safe app-specific actions like opening URLs, revealing files, and opening VS Code paths

### 6. n8n layer

- `n8n` is included as an optional workflow automation service.
- It is not preloaded with custom workflows yet.
- I configured the container and persistent storage, plus basic auth env support.
- You use it when you want drag-and-drop automations around this platform, like:
  - webhook -> create task in FastAPI
  - schedule -> call report endpoint -> send elsewhere
  - Telegram/admin action -> trigger external systems

## What is included

- FastAPI backend with health checks and interactive OpenAPI docs
- Telegram bot built with `python-telegram-bot`
- Ollama-first LLM routing with Gemini Flash fallback
- SQLite persistence through SQLAlchemy for tasks, agents, memory, and reports
- Redis-backed Celery workers, beat scheduling, and Flower monitoring
- Basic-auth-protected agent API for registration, heartbeats, task claiming, and result callbacks
- Direct device control API plus browser control panel at `/control`
- Optional `n8n` container in the same `docker-compose.yml`
- Installable desktop agent foundation in [`agent_daemon/`](agent_daemon/)
- GitHub Actions workflows for Windows and macOS agent artifact builds

## Quick start

1. Edit `.env` and set at least:
   - `TELEGRAM_BOT_TOKEN`
   - `TELEGRAM_ALLOWED_USER_IDS`
   - `AGENT_API_SHARED_KEY`
   - `GEMINI_API_KEY` if you want cloud fallback
   - `OLLAMA_MODEL` to your preferred default model
   - `OLLAMA_MODELS` to the full list you want pre-pulled on first boot
   - `N8N_BASIC_AUTH_PASSWORD`
2. Start everything:

```bash
docker compose up -d
```

`docker-compose up -d` works too when that binary is installed.

3. Open the main services:
   - API docs: `http://YOUR_VPS_IP:8000/docs`
   - Health: `http://YOUR_VPS_IP:8000/health`
   - Control panel: `http://YOUR_VPS_IP:8000/control`
   - Flower: `http://YOUR_VPS_IP:5555`
   - n8n: `http://YOUR_VPS_IP:5678`

The first Ollama startup can take several minutes because [`docker/ollama/entrypoint.sh`](docker/ollama/entrypoint.sh) automatically pulls every model in `OLLAMA_MODELS`.

## Ollama and models

Recommended starter setup:

```env
OLLAMA_MODEL=qwen2.5:3b
OLLAMA_MODELS=qwen2.5:3b,qwen2.5-coder:7b,gemma2:2b
```

Suggested usage:

- `qwen2.5:3b` or `qwen2.5:7b` for Telegram chat, task coordination, and daily reports
- `qwen2.5-coder:7b` for code-focused work from Continue or other dev tools
- `gemma2:2b` as a smaller backup local model

Important note:

- Continue is separate from this platform.
- Continue can point directly at your Ollama server on `http://YOUR_VPS_IP:11434`.
- That means Continue can use a code model like `qwen2.5-coder:7b` even while the Telegram bot is using `qwen2.5:3b`.
- You do not need the Telegram bot and Continue to share the same active model unless you want them to.

## Runtime model control

### From Telegram

- `/models`
- `/usemodel ollama qwen2.5:3b`
- `/usemodel ollama qwen2.5-coder:7b`
- `/usemodel gemini gemini-1.5-flash`

### From your app

List models:

```bash
curl http://YOUR_VPS_IP:8000/api/v1/models
```

Switch active model:

```bash
curl -X POST http://YOUR_VPS_IP:8000/api/v1/models/select \
  -H "Content-Type: application/json" \
  -d '{"provider":"ollama","model":"qwen2.5:3b","auto_pull":true}'
```

Example response:

```json
{
  "provider": "ollama",
  "model": "qwen2.5:3b"
}
```

## Telegram usage

Send plain messages for normal chat with memory, or use task commands:

- `start task office-pc: clean temp files`
- `start task cloud-worker: shell: uname -a`
- `check status`
- `check status tsk_123456789abc`
- `continue task tsk_123456789abc add more logging before rerun`
- `/models`
- `/usemodel ollama qwen2.5:3b`

Conversation history is stored in SQLite and reused per Telegram user session.

## Agent API

The agent routes are protected with HTTP Basic auth:

- Username: `AGENT_BASIC_AUTH_USERNAME`
- Password: `AGENT_API_SHARED_KEY`

Example auth header on Linux:

```bash
AUTH=$(printf 'agent:change-me-now' | base64)
```

Register an agent:

```bash
curl -X POST http://YOUR_VPS_IP:8000/api/v1/agents/register \
  -H "Authorization: Basic $AUTH" \
  -H "Content-Type: application/json" \
  -d '{"name":"office-pc","capabilities_json":["shell","desktop"],"metadata_json":{"os":"windows"}}'
```

Claim work:

```bash
curl -X POST http://YOUR_VPS_IP:8000/api/v1/agent-tasks/claim \
  -H "Authorization: Basic $AUTH" \
  -H "Content-Type: application/json" \
  -d '{"agent_name":"office-pc","allow_unassigned":true}'
```

Post a result:

```bash
curl -X POST http://YOUR_VPS_IP:8000/api/v1/agent-tasks/tsk_123456789abc/result \
  -H "Authorization: Basic $AUTH" \
  -H "Content-Type: application/json" \
  -d '{"agent_name":"office-pc","status":"completed","result_text":"done"}'
```

## Device control

Open the browser control panel:

```text
http://YOUR_VPS_IP:8000/control
```

What it can do right now:

- live screenshot polling for a near-real-time device view
- click directly on the screenshot to send remote mouse clicks
- move, click, and drag the mouse
- type text, press keys, and send hotkeys
- browse directories and read file contents
- list processes and kill a process by PID
- list windows and focus a matching window
- launch apps
- run safe app actions:
  - open URL in browser
  - reveal path in file manager
  - open a path in VS Code

Main control endpoints:

- `POST /api/v1/control/commands`
- `GET /api/v1/control/commands`
- `GET /api/v1/control/commands/{command_id}`
- `GET /api/v1/control/commands/{command_id}/artifact`

Agent-side control endpoints:

- `POST /api/v1/agent-control/claim/{agent_name}`
- `POST /api/v1/agent-control/{command_id}/result`

## Desktop agent installers

The agent is now structured for one-click installation instead of "install Python and run a script" as the primary path.

### End-user flow

- Windows user runs `PersonalAIOpsAgentSetup.exe`
- macOS user installs `PersonalAIOpsAgent.pkg`
- the setup window opens on first launch
- the user enters:
  - server URL
  - agent name
  - shared key
- the app saves config, validates the VPS connection, enables auto-start, and starts the daemon

### What the installed agent does

- registers with the VPS
- sends heartbeats
- claims tasks
- executes `shell:` and `powershell:` tasks
- executes direct remote-control commands
- captures screenshots for live view
- performs mouse and keyboard automation
- browses files and reads file content
- lists processes and windows
- launches apps and runs safe app-specific actions
- posts results back
- starts automatically on login

### Build the Windows installer

From a Windows machine:

```powershell
cd agent_daemon
powershell -ExecutionPolicy Bypass -File packaging\windows\build.ps1
```

Outputs:

- `agent_daemon\dist\PersonalAIOpsAgent.exe`
- `agent_daemon\dist\PersonalAIOpsAgentSetup.exe` if Inno Setup is installed

### Build the macOS installer

From a macOS machine:

```bash
cd agent_daemon
bash packaging/macos/build.sh
```

Outputs:

- `agent_daemon/dist/PersonalAIOpsAgent.app`
- `agent_daemon/dist/PersonalAIOpsAgent.pkg`

Optional DMG wrapper:

```bash
cd agent_daemon
bash packaging/macos/create-dmg.sh
```

Output:

- `agent_daemon/dist/PersonalAIOpsAgent.dmg`

### GitHub Actions builds

The repo now includes:

- [`build-agent-windows.yml`](.github/workflows/build-agent-windows.yml)
- [`build-agent-macos.yml`](.github/workflows/build-agent-macos.yml)

These workflows can build and upload agent artifacts automatically from GitHub Actions:

- Windows `.exe`
- Windows setup `.exe` when Inno Setup is available
- macOS `.pkg`
- macOS `.dmg`

## Notes

- SQLite runs from `./data/ai_ops.db` inside the stack using WAL mode for a simpler self-hosted setup.
- Celery Beat schedules daily reports and periodic health snapshots automatically.
- If `DEFAULT_REPORT_CHAT_ID` is set, daily reports are also pushed to Telegram.
- n8n starts with container-level config only. You still create your own workflows in the n8n UI after login.
- The packaged desktop agent stores its config and logs in the current user's local application data directory.
- GUI automation and screenshots require the agent machine to have an active desktop session.
